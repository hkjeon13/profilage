# Profilage UI/UX 코드 리뷰

- 작성일: 2026-07-10
- 검토 범위: 홈 검색, 검색 실패, 기업 프로필/하위 페이지 상태, 기업 비교 선택·조회·공유, 검색 체감 성능, 모바일 반응형, 주요 모달·탭·데이터 시각화의 접근성 코드
- 검토 방식: 로컬 앱을 1440×1000 및 390×844 뷰포트에서 실행해 현재 화면을 캡처한 뒤, `app/static/index.html`, `profile.html`, `compare.html`, `app.js`, `profile-page-5.js`, `compare-page.js`, `styles.css`를 교차 검토했다.

## 요약

현재 UI는 홈 검색의 목적이 명확하고, 프로필 영역의 시각적 톤과 데이터 출처 표기는 비교적 일관적이다. 반면 핵심 사용자 여정인 **검색 → 프로필 → 비교**가 실패·빈 상태에서 쉽게 끊긴다. 특히 검색 실패 후 화면이 거의 비어 보이고, 프로필 오류 및 일부 정상 하위 페이지에서도 로딩 문구가 남으며, 비교 대상 추가·조회·공유 상태가 실제 데이터와 다르게 보일 수 있는 문제가 전환과 데이터 신뢰를 크게 방해한다.

우선순위는 다음과 같다.

1. 검색/프로필 오류 상태를 복구 가능한 단일 상태로 재설계한다.
2. 재무·공시 하위 페이지와 비교 조회의 부분 실패를 정확한 상태로 표시한다.
3. 비교 대상 선택, 최대 개수, 공유 URL의 상태 계약을 통일한다.
4. 검색 요청 수와 취소 정책을 개선해 체감 대기시간과 불필요한 요청을 줄인다.
5. 모달 포커스, 탭 의미 구조, 표·차트 접근성, 터치 타깃을 공통 컴포넌트 수준에서 보완한다.
6. 헤드라인 줄바꿈, 링크 대비, 빈 공간 등 시각적 마감을 정리한다.

## 우선순위별 개선안

### P0. 검색 실패를 복구 가능한 결과 상태로 만든다

**근거 코드**

- `app/static/app.js:354-388`에서 검색 시작 시 즉시 `is-idle`을 제거한다.
- `app/static/styles.css:97-105`는 비유휴 상태에서 헤드라인, 예시 기업, 데이터 출처, 푸터를 모두 숨긴다.
- `app/static/app.js:384-388`의 실패 처리는 상태 텍스트 설정과 결과 삭제만 수행한다.
- `app/static/index.html:68-73`에는 한 줄 상태와 빈 결과 컨테이너만 있다.

**문제**

- 오류 원인과 복구 방법이 없다.
- 오류가 발생해도 사용자가 다른 기업을 선택할 수 있는 예시와 출처 정보까지 사라진다.
- 검색 실패와 검색 결과 없음이 시각적으로 거의 같은 빈 화면으로 보인다.

**권장 변경**

- 결과 영역에 `loading`, `success`, `empty`, `error`의 명시적인 뷰 상태를 둔다.
- 오류 카드에는 `검색을 다시 시도`, `검색어 수정`, `예시 기업 선택`을 제공한다.
- 오류 메시지는 “기업 정보를 불러오지 못했습니다. 잠시 후 다시 시도해주세요.”처럼 행동 중심 문구로 바꾸고, 기술 상세는 접을 수 있는 보조 정보로 분리한다.
- 실패 시 데이터 출처와 예시 기업을 유지하거나 결과 영역 안에 축약 형태로 다시 노출한다.
- 검색 버튼과 입력에 `aria-busy`를 연결하고, 결과 전체가 아닌 짧은 상태 메시지만 라이브 영역으로 알린다.

**완료 기준**

- 실패 후 한 번의 클릭 또는 Enter로 같은 검색을 재시도할 수 있다.
- 입력한 검색어와 URL의 `q` 값이 유지된다.
- 키보드/스크린리더 사용자에게 로딩과 실패가 한 번씩만 전달된다.

### P0. 프로필 오류 시 로딩 UI를 모두 종료한다

**근거 코드**

- 초기 로딩 문구가 `app/static/profile.html:36-60`의 히어로, 기본정보, 상세 영역 세 곳에 존재한다.
- `app/static/profile-page-5.js:806-817`의 `renderError()`는 히어로와 상세 영역만 바꾸고 `#profile-basic-card`를 갱신하지 않는다.
- 같은 함수가 `${message}`를 `innerHTML`에 직접 넣어 예상하지 못한 마크업이 렌더링될 위험도 있다.
- `.empty-state`의 고정 최소 높이(`app/static/styles.css:1460-1467`)와 프로필 상세 최소 높이(`app/static/styles.css:1844-1851`)가 모바일의 큰 공백을 만든다.

**문제**

- “열 수 없습니다”와 “불러오는 중입니다”가 동시에 보여 상태가 모순된다.
- 같은 오류가 히어로와 상세 영역에서 중복된다.
- 오류 상태에서도 `DATA` 평가 카드가 남아 오류 설명과 겹친다.
- 모바일에서는 복구 행동이 첫 화면 밖으로 밀린다.

**권장 변경**

- `setProfileState('loading' | 'ready' | 'error')`처럼 프로필 전체를 원자적으로 전환하는 상태 함수를 만든다.
- 오류 시 기본정보 placeholder와 평가 카드를 숨기고, 상단 그리드를 하나의 오류 카드로 합친다.
- 오류 카드는 `다시 시도`, `검색 결과로 돌아가기`, `다른 기업 검색`을 제공한다.
- 오류 메시지는 `textContent` 또는 `escapeHtml()`을 통해 렌더링한다.
- 오류/빈 상태의 최소 높이는 모바일에서 콘텐츠 높이에 맞추고, 의미 없는 고정 공백을 제거한다.

**완료 기준**

- 오류 화면 어디에도 “불러오는 중” 문구나 skeleton이 남지 않는다.
- 390×844 첫 화면 안에서 오류 설명과 최소 하나의 복구 CTA가 보인다.
- 같은 오류 문구가 한 번만 노출된다.

### P0. 재무·공시 하위 페이지에서도 로딩 상태를 완전히 종료한다

**근거 코드**

- `renderProfileSkeleton()`은 히어로에 `is-loading`을 추가하고 프로필 상세를 skeleton으로 교체한다(`app/static/profile-page-5.js:782-804`).
- `loadProfile()`의 `financials` 및 `disclosures` 분기는 각 하위 페이지를 렌더링한 직후 반환한다(`app/static/profile-page-5.js:2728-2763`).
- 히어로 로딩 해제와 `#profile-basic-card` 갱신은 일반 프로필 경로의 `renderCompanyDetail()`에서만 수행된다(`app/static/profile-page-5.js:2651-2667`).

**문제**

- API가 정상 응답해도 재무제표 또는 공시 하위 페이지의 기본정보 카드가 계속 “불러오는 중입니다”로 남을 수 있다.
- 오류 화면뿐 아니라 정상 “더보기” 여정에서도 완료 상태와 로딩 상태가 동시에 보인다.

**권장 변경**

- 하위 페이지 렌더링 전에 공통 `completeProfileHeader(info, outline, listed, crno)`를 호출해 히어로, 로고, 기본정보 카드, `is-loading`을 함께 갱신한다.
- 프로필 공통 헤더와 상세 본문 상태를 분리하되, 각 경로가 반드시 `loading → ready/error`로 종료되도록 상태 전이를 한곳에서 관리한다.
- 재무·공시 하위 페이지를 직접 URL로 열었을 때도 동일한 완료 처리가 실행되는지 검증한다.

**완료 기준**

- `/profile?...&view=financials`와 `/profile?...&view=disclosures`에서 로딩 placeholder와 skeleton이 남지 않는다.
- 일반 프로필, 재무, 공시 경로가 동일한 기업명·기본정보·뒤로가기 상태를 보여준다.

### P0. 비교 조회의 전체·부분 실패를 빈 상태나 성공 상태로 위장하지 않는다

**근거 코드**

- 비교 페이지는 기업별 `loadCompany()` 오류를 모두 `null`로 바꾼 뒤 제거한다(`app/static/compare-page.js:523-537`).
- 개별 오류를 이미 삼켰기 때문에 하단 `catch`의 오류 UI는 일반적인 기업 조회 실패에서 실행되기 어렵다(`app/static/compare-page.js:538-540`).

**문제**

- 선택한 기업이 모두 실패하면 실제 원인은 조회 오류인데 “비교할 기업을 더 추가해주세요”가 표시된다.
- 일부만 실패하면 실패한 기업과 원인에 대한 설명 없이 남은 기업만 정상 비교 결과처럼 표시된다.
- 금융/기업 데이터 비교에서 조용한 누락은 사용자의 판단을 왜곡할 수 있다.

**권장 변경**

- `Promise.allSettled()` 또는 성공/실패를 함께 반환하는 로더를 사용해 선택한 기업별 결과를 보존한다.
- 전체 실패는 별도의 오류 화면과 `다시 시도`를 제공한다.
- 부분 실패는 비교표 위에 누락 기업명, 실패 개수, 재시도 행동을 표시하고 사용자가 계속 비교할지 결정하게 한다.
- 빈 선택, 1개 선택, 전체 실패, 부분 실패를 서로 다른 뷰 상태로 정의한다.

**완료 기준**

- 선택 수와 실제 로드 수가 다르면 그 차이가 화면과 `role="status"`에 명시된다.
- 실패한 기업을 개별 또는 전체 재시도할 수 있다.
- 조회 실패가 “기업을 더 추가해주세요” 문구로 표시되지 않는다.

### P1. 비교 선택을 일회성 버튼이 아닌 지속 상태로 보여준다

**근거 코드**

- 검색 결과의 `비교 추가`는 `app/static/app.js:233-247`에서 텍스트를 `추가됨`으로 바꾸고 비활성화하는 것으로 끝난다.
- 비교 페이지는 2개 미만일 때 검색 화면 링크만 제공한다(`app/static/compare-page.js:493-502`).
- 검색 화면의 최대 기업 수는 4개(`app/static/app.js:9-10`)지만 비교 페이지는 5개(`app/static/compare-page.js:1-2`)로 계약이 다르다.
- 이미 4개가 저장된 검색 화면에서는 새 기업이 `.slice(0, 4)`로 제외되어도 버튼이 무조건 `추가됨`으로 바뀐다(`app/static/app.js:240-247`).
- 프로필의 비교 이벤트 코드는 존재하지만 실제 요약 카드에는 대응하는 `data-compare-add`/`data-compare-link` 요소가 없다(`app/static/profile-page-5.js:2274-2294`, `2541-2553`).

**문제**

- 기업을 추가한 뒤 어디에서 비교를 시작하는지 알기 어렵다.
- 현재 몇 개가 선택되었는지, 어떤 기업이 담겼는지, 제한이 몇 개인지 보이지 않는다.
- 화면별 최대 선택 개수가 달라 예측 가능성이 떨어진다.
- 선택 한도에 도달하면 저장되지 않은 기업도 “추가됨”으로 표시될 수 있다.

**권장 변경**

- 검색 결과 하단 또는 상단에 고정형 비교 트레이를 두고 선택 기업, 개수, 삭제, `2개 기업 비교하기` CTA를 제공한다.

### P1. 공유 비교 URL을 수신자의 로컬 비교함과 분리한다

**근거 코드**

- `requestedCrnos()`는 URL의 `crno`와 수신자 브라우저의 `localStorage` 항목을 항상 합친다(`app/static/compare-page.js:193-198`).

**문제**

- 공유자가 의도한 비교 조합과 수신자가 보는 조합이 달라져 링크의 재현성과 신뢰가 깨진다.

**권장 변경**

- URL에 기업이 없을 때만 `localStorage`를 fallback으로 사용한다.
- 공유 링크로 연 비교와 개인 비교함 병합은 자동 수행하지 말고 사용자가 선택할 수 있는 별도 행동으로 제공한다.

**완료 기준**

- 동일한 공유 URL은 새 브라우저와 기존 비교함이 있는 브라우저에서 같은 기업 조합을 표시한다.
- 공유 링크를 열어도 개인 비교함이 조용히 변경되지 않는다.

### P1. 검색 요청 수와 취소 정책을 개선한다

**근거 코드**

- 한 검색어에 14개 접미사 후보를 만들어 KRX 요청을 병렬 실행하고, 기업개요 요청에서는 최대 1,000개를 요청한다(`app/static/app.js:251-271`, `329-344`).
- 첫 결과 20개를 렌더링할 때 법인등록번호별 KRX 보강 요청이 최대 20건 추가된다(`app/static/app.js:175-203`, `205-225`).
- `searchState.token`은 오래된 응답의 렌더링만 막을 뿐 진행 중인 네트워크 요청을 취소하지 않는다(`app/static/app.js:354-388`).

**문제**

- 첫 검색에 초기 요청 최대 15건과 결과 보강 최대 20건, 즉 최대 35건 수준의 요청이 발생할 수 있다.
- 사용자가 검색을 연속 실행하면 이전 요청이 계속되어 새 결과가 늦어지고 서버·외부 API 제한에 불필요한 부담을 준다.
- 검색 버튼과 예시 기업 버튼이 로딩 중에도 활성화되어 중복 요청을 만들기 쉽다.

**권장 변경**

- 기업개요와 상장정보를 서버의 통합 검색/배치 엔드포인트에서 한 번에 조회한다.
- 접미사 확장은 서버 검색 정책으로 이동하거나 실제로 필요한 짧은 검색어에만 제한한다.
- `AbortController`로 이전 검색과 결과 보강 요청을 취소한다.
- 로딩 중 입력은 유지하되 제출 버튼과 예시 버튼의 중복 실행을 제어하고 명확한 진행 상태를 제공한다.
- 검색 결과 수, 최초 결과 표시 시간, 총 요청 수를 관측 지표로 남긴다.

**완료 기준**

- 새 검색이 시작되면 이전 검색의 네트워크 요청이 취소된다.
- 첫 결과 표시를 위해 필요한 프론트엔드 API 왕복 수가 고정된 소수로 제한된다.
- 느린 네트워크에서도 최신 검색어의 상태만 화면에 표시된다.

- 카드 전체를 클릭 가능하게 만들지 않을 경우 카드 커서를 기본값으로 되돌린다.
- 카드 전체를 클릭 가능하게 만들 경우 하나의 명확한 primary link를 stretched-link 패턴으로 확장하되, 내부의 비교 버튼과 포커스 순서를 충돌 없이 유지한다.
- 결과는 제목이 있는 `<section>`과 목록 구조로 제공하고, 동적 낭독은 결과 전체가 아니라 결과 수와 상태를 담은 짧은 `role="status"`에만 적용한다.

### P2. 사용자 문구와 상태 언어를 통일한다

**문제**

- `Loading`, `Error`, `DATA`와 한국어가 한 화면에 혼재한다.
- `요청에 실패했습니다.`는 사용자가 취할 행동을 설명하지 않는다.
- 비상장과 데이터 확인 실패를 `비상장/확인 필요` 하나로 묶어 의미가 모호하다.

**권장 변경**

- 사용자 상태 문구를 `불러오는 중`, `정보를 불러오지 못함`, `데이터 기반 평가`처럼 한국어 중심으로 통일한다.
- 데이터 부재, 비상장, 조회 실패, 권한/서버 문제를 서로 다른 상태와 설명으로 구분한다.
- 기술 오류는 사용자 문구와 분리하고 로깅용 식별자만 보조로 제공한다.

### P2. 모션 감소 설정과 재무 차트의 대체 표현을 제공한다

**근거 코드**

- 검색 skeleton과 공시 요약 로딩 표면은 지속적인 shimmer 애니메이션을 사용하지만 `prefers-reduced-motion` 예외가 없다(`app/static/styles.css:495-517`, `1282-1314`).
- 재무 추이 차트는 컨테이너에 일반적인 `role="img"`와 라벨만 있고, 시계열 값을 읽을 수 있는 표·목록 대안이나 키보드 탐색을 제공하지 않는다(`app/static/profile-page-5.js:1379-1427`).
- 여러 계열은 선 색상과 범례 색상으로 구분된다.

**권장 변경**

- `@media (prefers-reduced-motion: reduce)`에서 shimmer와 불필요한 전환을 중지한다.
- 재무 추이 차트 아래에 연도별 값을 담은 접근 가능한 표 또는 `데이터 표 보기` 토글을 제공한다.
- 선 종류, 점 마커, 텍스트 라벨을 함께 사용해 색상만으로 계열을 구분하지 않는다.
- 차트의 접근 가능한 이름에 선택 계정과 기간을 포함하고 최신값·증감 요약을 텍스트로 제공한다.

**완료 기준**

- 모션 감소 환경에서 반복 애니메이션이 실행되지 않는다.
- 차트를 보지 않고도 모든 연도별 수치와 계열을 확인할 수 있다.
- 200% 확대와 키보드만으로 차트 대체 정보를 사용할 수 있다.

## 잘된 부분

- 검색 입력에 실제 `<label>`이 연결되어 있고 검색 버튼에 접근 가능한 이름이 있다(`app/static/index.html:32-53`).
- 검색 로딩 skeleton이 `aria-hidden="true"`여서 불필요한 낭독을 줄인다(`app/static/app.js:99-116`).
- 프로필 기본정보는 `<dl>` 구조를 사용해 라벨과 값의 관계가 명확하다(`app/static/profile-page-5.js:2612-2638`).
- 주가 차트는 키보드 포커스와 방향키 탐색 코드를 포함하고 있어 시각화 접근성의 기반이 있다.
- 모바일에서 주요 그리드가 단일 열로 전환되고 비교 빈 상태 CTA가 전체 폭으로 확장된다.
- 데이터 출처, 갱신 시각, 캐시 만료 등 신뢰 정보를 표시하려는 구조가 이미 존재한다.

- 모든 오류 상태에서 재시도 또는 이전 단계 복귀가 한 번의 조작으로 가능한가?

- 프로필 섹션 내비게이션의 모든 링크가 실제로 존재하고 의미가 일치하는 섹션으로 이동하는가?

## 검토 한계

- 현재 로컬 실행 환경에는 `OPEN_API_DECODING_KEY`/`OPEN_API_ENCODING_KEY`가 설정되어 있지 않아 외부 데이터 요청이 실패했다. 따라서 실제 검색 성공 결과, 정상 기업 프로필/하위 페이지, 정상 비교표, 공시 요약/뷰어, 주주 상세의 완성 화면을 이번 실행에서 캡처하지 못했다.
- 성공 상태의 정보 우선순위와 실제 데이터 길이에 따른 레이아웃은 코드 정적 검토에 기반한 제안이며, 실제 API가 연결된 환경에서 추가 시각 검증이 필요하다.
- 색 대비, 스크린리더 낭독 순서, 키보드 포커스 트랩은 화면과 코드에서 확인한 위험이다. WCAG 준수 여부를 확정하려면 자동 검사와 실제 키보드/스크린리더 테스트가 별도로 필요하다.
