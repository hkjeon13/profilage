# 기업 정보 기능 확장 아이디어

작성일: 2026-07-07

## 목적

Profilage의 현재 코드와 연결된 API 범위를 기준으로, 기업 정보 제품에서 추가하면 좋은 기능을 두 범주로 나누어 정리한다.

1. 현재 코드에 이미 연결된 API 또는 현재 API key 종류로 구현 가능한 기능
2. 추가 데이터 수집, 배치 저장, 또는 신규 외부 API 연동이 필요한 기능

주의: 이 문서는 코드에 존재하는 설정명과 서비스 구현을 기준으로 작성했다. 로컬 워크스페이스에는 `.env` 파일이 없어 실제 운영 key 값이나 활성 상태는 확인하지 않았다.

## 현재 연결된 데이터 소스

| 데이터 소스 | 설정명 | 현재 활용 범위 | 비고 |
| --- | --- | --- | --- |
| 금융위원회 기업기본정보 | `OPEN_API_DECODING_KEY` 또는 `OPEN_API_ENCODING_KEY` | 기업개요, 계열회사, 연결대상 종속기업, KRX 상장종목정보 | 공공데이터포털 설명상 일 1회 갱신, 실시간 데이터가 아님 |
| DART OpenAPI | `DART_API_KEY` | DART 고유번호, 회사개황, 공시목록, 재무제표 주요계정, 정기보고서 주요정보, 공시 원문 문서 | 재무, 주주, 임원, 직원, 배당, 감사의견 등 확장 여지가 큼 |
| SearchAPI Google Finance | `SEARCHAPI_API_KEY` | 현재가, 기간별 주가 차트, Google Finance 응답 기반 시장 데이터 | SearchAPI 문서상 뉴스, 주요 이벤트, 재무 등도 응답 범위에 포함될 수 있으나 현재 파서는 주가 중심 |
| OpenAI | `OPENAI_API_KEY`, `OPENAI_MODEL` | 공시 원문 요약, 기업 프로필 요약 | 원문 길이 제한과 캐시 전략 필요 |
| 공정위 기업집단 API | `BUSINESS_GROUP_SERVICE_KEY`, `EGROUP_SERVICE_KEY`, 또는 `OPEN_API_DECODING_KEY` | 상위 기업집단/소속회사 동기화, 주주 검색 인덱싱 코드 존재 | `DATABASE_URL`이 있어야 동기화/검색 가치가 커짐 |
| Postgres 저장소 | `DATABASE_URL` | API 응답 캐시, 기업집단, 주주 보유 인덱스 | 히스토리/스냅샷 기능의 핵심 기반 |
| Valkey 캐시 | `VALKEY_URL` | JSON 응답 캐시 | 실시간성/호출비용 최적화 |

## 현재 API key 범위로 바로 추가하기 좋은 기능

### 1. 기업 한눈에 요약 카드 고도화

현재 가능한 데이터:

- 금융위원회 기업개요: 대표자, 주소, 설립일, 종업원 수, 사업자등록번호, 법인등록번호
- DART 회사개황: CEO, 결산월, 홈페이지, 전화번호, 상장/비상장 구분
- DART 재무제표: 최근 사업연도/분기 주요 계정
- DART 정기보고서 주요정보: 최대주주, 배당, 감사의견, 재무비율
- 기존 기업 프로필 요약 캐시

추가 기능:

- 신규 AI 호출 없이 기존 `get_company_info` 응답을 핵심 카드로 재구성
- 최근 재무 상태: 매출, 영업이익, 순이익, 자산, 부채, 자본
- 지배구조 요약: 최대주주명, 지분율, 특수관계 여부
- 배당/감사 의견/부채비율 같은 핵심 리스크 배지
- 기존 AI 요약이 이미 캐시되어 있으면 보조 설명으로만 재사용
- 데이터 기준일과 출처를 카드 하단에 명확히 표시

우선순위: 높음

이유: 이미 대부분의 원천 데이터가 `get_company_info`와 DART insight 계층에 모여 있다. 별도 AI 호출을 늘리지 않고도 사용자 체감 가치가 크다.

### 2. 공시 이벤트 타임라인

현재 가능한 데이터:

- `get_dart_disclosures`
- `normalize_disclosure_events`
- 공시 유형 분류: 정기보고서, 지분/최대주주, 임원, 자본, 배당, 감사/회계, 기타
- viewer URL
- 공시 원문 요약 API

추가 기능:

- 최근 공시를 날짜순 타임라인으로 표시
- 유형별 필터 제공
- 주요 공시는 “요약 보기” 버튼 제공
- 정정공시, 지분공시, 감사보고서 등을 시각적으로 구분
- 공시 이벤트를 기업 프로필 첫 화면의 “최근 변화” 섹션으로 노출

우선순위: 높음

이유: 현재 코드에 공시 분류와 요약 기반이 이미 있다. 주가/리스크/요약 기능의 공통 기반이 된다.

### 3. 주가 차트와 공시 마커 결합

현재 가능한 데이터:

- SearchAPI Google Finance 주가 차트
- DART 공시 날짜와 viewer URL
- 공시 이벤트 분류

추가 기능:

- 차트 위에 공시 마커 표시
- 마커 hover/click 시 공시명, 날짜, 유형, 요약 버튼 표시
- 공시일이 비거래일이면 다음 거래일 포인트에 매핑
- 장마감 후 공시는 다음 거래일 이벤트로 표시하는 정책 적용

우선순위: 높음

이유: 신규 데이터 없이 “주가가 왜 움직였는지”를 설명하는 경험을 만들 수 있다.

### 4. 규칙 기반 재무/리스크 신호

현재 가능한 데이터:

- DART 재무비율
- DART 재무제표 주요계정
- 감사의견
- 배당 정보
- 계열회사/종속기업 수

추가 기능:

- 부채비율 높음
- 영업이익률 음수
- ROE/ROA 낮음 또는 음수
- 감사의견이 적정이 아님
- 계열/종속기업 구조 복잡도 높음
- 배당성향 과도 또는 배당 없음
- 단일 보고서로 판단 가능한 신호와 히스토리가 필요한 신호를 분리 표시

우선순위: 높음

주의: 투자 판단처럼 보이지 않도록 “데이터 기반 체크포인트”로 표현한다.

### 5. 기업 비교 화면 고도화

현재 가능한 데이터:

- `compare` 페이지
- `get_company_info`
- 재무, 주가, 기본정보, 관계회사 수
- localStorage 기반 비교함

추가 기능:

- 비교 기업별 핵심 지표 표: 매출, 영업이익, 순이익, 자산, 부채비율, ROE, 최대주주 지분율
- “우위/열위”를 색으로 표시하되, 해석 문구는 보수적으로 제공
- 비교 기업의 공시 이벤트 빈도 비교
- 상장사만 주가 수익률 비교
- 비상장사는 주가 컬럼을 자동 숨김

우선순위: 중상

이유: 이미 비교 흐름이 있고, 현재 데이터만으로도 B2B 리서치 도구처럼 보일 수 있다.

### 6. 주주/지배구조 탐색

현재 가능한 데이터:

- DART 최대주주/소액주주 주요정보
- 공정위 기업집단 소속회사 동기화
- `shareholders/search`
- DART holdings 인덱싱 코드

추가 기능:

- 최대주주 Top N 표시
- 동일 주주가 보유한 다른 기업 검색
- 기업집단 내 소속회사와 지분 관계 탐색
- “이 기업은 어느 그룹과 연결되어 있는가” 섹션
- 오너/법인/기관 주주를 이름 규칙으로 1차 분류

우선순위: 중상

전제: `DATABASE_URL` 기반 인덱싱을 운영 배치로 안정화해야 한다.

### 7. 임원/직원 정보 섹션

현재 가능한 데이터:

- DART 임원 현황
- DART 직원 현황
- DART 정기보고서 기준 연도/보고서 코드

추가 기능:

- 주요 임원 목록
- 등기임원/미등기임원 구분
- 성별/근속연수/보수 총액 등 공개 항목 요약
- 직원 수와 평균 근속연수 변화
- 대표이사, 사외이사, 감사 관련 공시 이벤트 연결

우선순위: 중간

주의: 개인 정보처럼 느껴질 수 있는 항목은 공개 공시 출처와 기준일을 명확히 표시한다.

### 8. 기존 AI 요약 사용 정책 정리

현재 가능한 데이터:

- OpenAI Responses API 호출 코드
- DART 공시 원문 API
- 기업 프로필 요약 캐시

추가 기능:

- AI 요약은 사용자가 명시적으로 요청하거나 캐시가 없고 요약 섹션을 열 때만 호출
- 이미 생성된 기업 프로필 요약은 카드/공시 화면에서 재사용
- “사업”, “재무”, “지배구조”, “최근 이슈”, “주의할 점” 같은 구조는 신규 호출보다 기존 요약 응답 스키마 정리에 우선 적용
- 요약 근거가 된 데이터 기준일 표시
- 요약 실패 또는 미호출 상태에서는 규칙 기반 카드만 표시

우선순위: 중간

주의: OpenAI 호출 비용과 지연시간을 고려해, “요약을 더 만들기”보다 기존 요약 캐시 재사용과 호출 조건 제한을 우선한다.

## 현재 API로 가능하지만 추가 수집/저장이 필요한 기능

아래 기능은 신규 외부 API가 반드시 필요한 것은 아니지만, 단건 조회만으로는 품질이 낮다. `DATABASE_URL` 기반 스냅샷, 배치 수집, 정규화 테이블이 필요하다.

### 1. 변화 탐지

필요한 저장 데이터:

- 연도/분기별 재무제표 주요계정
- 연도/분기별 재무비율
- 연도별 최대주주 목록
- 연도별 임원 목록
- 연도별 배당 정보
- 기준일별 계열회사/종속기업 목록

가능한 신호:

- 매출/영업이익/순이익 급증락
- 영업이익 적자 전환
- 부채비율 급등
- 최대주주 변경
- 주요 임원 변경
- 배당 중단 또는 급감
- 계열회사/종속기업 수 변화

권장 구현:

- `company_snapshots` 또는 기존 `company_data_groups`를 확장
- entity key는 `crno`, `corp_code`, `stock_code`를 모두 저장
- `snapshot_period`, `source`, `group_name`, `payload_hash`로 중복 방지
- 일별 데이터와 보고서별 데이터를 분리

### 2. 기업 검색 인덱스

필요한 저장 데이터:

- 금융위원회 기업개요 전체 또는 자주 조회되는 기업 subset
- DART corp code 전체 목록
- KRX 상장종목 목록
- 최근 검색/인기 검색

가능한 기능:

- 자동완성
- 오타/약칭 검색
- 법인명, 종목명, 영문명, 사업자번호 통합 검색
- 상장/비상장 필터
- 업종/지역 필터

권장 구현:

- Postgres trigram 또는 별도 search table
- 초기는 local DB + scheduled sync로 충분
- 나중에 Meilisearch/OpenSearch로 분리 가능

### 3. 기업집단/주주 네트워크 그래프

필요한 저장 데이터:

- 공정위 기업집단 목록
- 기업집단 소속회사
- DART 주요주주 holdings
- DART 임원/주요주주 보고서

가능한 기능:

- 그룹별 소속회사 그래프
- 동일 주주가 연결하는 기업 목록
- 상위 20개 그룹의 핵심 보유관계 탐색
- 회사 프로필에서 “연결된 기업” 추천

권장 구현:

- 현재 `business_groups`, `business_group_companies`, shareholder holdings 테이블을 먼저 운영화
- 동기화 API를 admin-only 또는 scheduled job으로 제한
- 검색 API는 public read-only로 제공

## 추가 API 또는 외부 데이터 연동이 필요한 기능

### 1. 뉴스/이슈 모니터링

추가 후보:

- SearchAPI Google News 또는 일반 Google 검색
- 네이버 검색 API 뉴스
- BIG KINDS
- 언론사 RSS/제휴 데이터

가능한 기능:

- 기업별 최신 뉴스
- 부정/중립/긍정 이슈 분류
- 공시와 뉴스의 날짜 연결
- 최근 30일 이슈 요약
- 임원/소송/리콜/실적 관련 키워드 알림

주의:

- 뉴스 저작권과 원문 저장 정책을 별도로 검토해야 한다.
- 요약은 링크/제목/스니펫 중심으로 하고 원문 전문 저장은 피하는 것이 안전하다.

### 2. 동종업계/피어 자동 추천

추가 후보:

- KRX 업종/지수 구성 데이터
- 통계청 KSIC 또는 표준산업분류 매핑 데이터
- NICE/KISVALUE/FnGuide 같은 유료 기업 데이터
- 공공데이터 업종별 기업/재무 데이터

가능한 기능:

- 같은 업종 기업 자동 추천
- 피어 대비 매출/이익/부채비율 비교
- 업종 평균 대비 위치 표시
- “비슷한 기업” 검색

주의:

- 현재 금융위원회 기업개요의 업종명만으로는 피어 추천 품질이 제한적이다.
- 상장사만 대상으로 먼저 구현하면 KRX 데이터로 정확도를 높일 수 있다.

### 3. 밸류에이션 지표

추가 후보:

- KRX 시가총액/상장주식수 데이터
- FnGuide, WiseReport, KISVALUE 등 재무/컨센서스 데이터
- SearchAPI Google Finance 응답 확장 파싱

가능한 기능:

- 시가총액
- PER, PBR, PSR
- EV/EBITDA
- 배당수익률
- 52주 고저가
- 컨센서스 대비 괴리율

주의:

- 현재 Google Finance 주가 응답만으로 모든 지표를 안정적으로 제공한다고 가정하면 위험하다.
- 투자 판단 기능처럼 보이지 않도록 출처와 계산식을 명확히 표시해야 한다.

### 4. 신용/부실 위험 데이터

추가 후보:

- NICE/KCB/KED 기업 신용정보
- 신용평가사 등급 공시
- 법원/회생/파산 관련 공개 데이터
- 조달청/국세/지방세 체납 공개 데이터가 있다면 별도 검토

가능한 기능:

- 신용등급
- 등급 변동
- 채무불이행/회생/파산 신호
- 거래처 리스크 체크

주의:

- 법적/계약적 제약이 크다.
- B2B 리스크 스코어로 제공하려면 면책 문구와 데이터 기준일이 필요하다.

### 5. ESG/제재/컴플라이언스

추가 후보:

- 한국ESG기준원 또는 민간 ESG 데이터
- 지속가능경영보고서 원문 수집
- 공정위 제재/심결 정보
- 금융감독원 제재 공시
- 조달/입찰 제한 데이터

가능한 기능:

- ESG 등급/보고서 링크
- 환경/사회/지배구조 이슈 요약
- 제재 이력
- 컴플라이언스 리스크 배지

주의:

- 출처마다 라이선스가 다르다.
- 자동 요약 시 원문 링크와 기준일을 반드시 함께 제공해야 한다.

### 6. 특허/기술/제품 데이터

추가 후보:

- KIPRIS 특허 검색 API
- 기업 홈페이지/IR 자료
- 조달청 나라장터 납품/계약 데이터

가능한 기능:

- 보유 특허 키워드
- 기술 분야 요약
- 주요 제품/서비스 자동 태깅
- 공공 조달 실적

주의:

- 법인명 매칭 품질이 핵심이다.
- 법인명, 사업자번호, 대표자, 주소 기반 entity resolution이 필요하다.

## 추천 우선순위

### 1단계: 신규 API 없이 제품 가치 올리기

1. 기업 한눈에 요약 카드 고도화
2. 공시 이벤트 타임라인
3. 주가 차트 공시 마커
4. 규칙 기반 리스크/체크포인트 카드
5. 비교 화면 핵심 지표 표 개선

이 단계는 현재 코드와 API key 범위에서 가능하다. 외부 API 계약 리스크가 낮고 구현 대비 체감 가치가 크다.

### 2단계: 저장/배치 기반 기능

1. 기업 검색 인덱스
2. 재무/주주/임원/배당 히스토리 스냅샷
3. 기업집단/주주 네트워크 인덱싱
4. 변화 탐지 카드
5. 모니터링/알림

이 단계는 `DATABASE_URL`과 scheduled job 운영이 핵심이다. 신규 API보다 데이터 파이프라인 안정성이 더 중요하다.

### 3단계: 신규 API 연동

1. 뉴스/이슈
2. 동종업계/피어
3. 밸류에이션
4. 신용/부실 위험
5. ESG/제재
6. 특허/기술

이 단계는 비용, 라이선스, 데이터 품질, 법적 표현 리스크를 먼저 검토해야 한다.
