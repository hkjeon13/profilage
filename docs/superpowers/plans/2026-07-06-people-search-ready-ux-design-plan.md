# People Search Ready UX Design Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 현재 기업 조회 UX를 해치지 않으면서, 추후 인물 조회 기능을 자연스럽게 붙일 수 있는 정보 구조, 내비게이션, 검색 결과, 상세 페이지 디자인 체계를 설계한다.

**Architecture:** 지금은 인물 조회 기능을 구현하지 않는다. 대신 화면 언어, 컴포넌트 명명, 검색/결과/상세 페이지 정보 구조를 `company-only`에서 `entity-ready`로 확장할 수 있게 설계하고, 실제 인물 API/데이터 모델은 별도 단계에서 추가한다.

**Tech Stack:** FastAPI static serving, vanilla JavaScript, CSS, existing company profile frontend, future DART executive/shareholder/person-related data sources, future person search API.

---

## 1. 배경

현재 Profilage는 기업 프로필 조회 중심이다.

현재 핵심 화면:

- 첫 화면 기업 검색
- 기업 검색 결과 리스트
- 기업 상세 프로필
- 기업 비교 화면

추후에는 사람, 즉 비즈니스 인물 조회 기능을 제공할 예정이다. 인물 조회는 단순한 이름 검색이 아니라, 기업과 연결된 사람의 역할, 소속, 경력, 주요 공시/보고서 등장 이력, 관련 기업을 찾는 업무형 검색 기능이 되어야 한다.

따라서 지금부터 디자인 체계는 아래 전환을 고려해야 한다.

```text
기업 검색 사이트
→ 기업·인물 검색 사이트
→ 비즈니스 엔티티 데이터 플랫폼
```

단, 현재 릴리스에서는 사용자에게 아직 제공하지 않는 기능을 과하게 노출하지 않는다.

---

## 2. 핵심 원칙

### 지금 당장 지켜야 할 원칙

- 현재 사이트는 여전히 기업 조회가 핵심이다.
- 인물 조회 UI를 미리 노출하되 비활성 기능처럼 보이게 만들지 않는다.
- 검색창, 결과 리스트, 상세 페이지 컴포넌트는 나중에 `기업/인물` 타입을 받을 수 있게 설계한다.
- 문구는 현재 기능을 과장하지 않는다.
- 코드 구현 시 `company`에 강하게 묶인 이름을 새로 늘리지 않고, 새 공통 컴포넌트는 `entity` 개념을 사용한다.

### 추후 인물 조회가 열릴 때 지켜야 할 원칙

- 사람 검색은 기업 검색과 같은 검색창에서 시작할 수 있어야 한다.
- 검색 결과에서 기업과 인물은 명확히 구분되어야 한다.
- 인물 상세 페이지는 이력서가 아니라 업무용 관계/역할 중심 프로필이어야 한다.
- 인물 정보는 개인정보처럼 보일 수 있으므로 출처, 기준일, 공개 데이터 여부를 더 명확히 표시한다.
- 동명이인 문제를 제품 UX에서 우선 처리해야 한다.

## 3. 정보 구조 방향

### 현재 유지할 구조

현재는 첫 화면에서 기업 조회 목적을 명확히 말한다.

추천 문구:

```text
기업 정보를 빠르게 찾고 비교하세요
```

현재 단계에서는 `인물 조회`를 headline에 넣지 않는다.

이유:

- 기능이 아직 없는데 기대를 만들면 신뢰가 떨어진다.
- B2B 데이터 플랫폼 느낌은 강화하되, 현 기능 범위를 넘지 않아야 한다.

### 추후 전환할 구조

인물 조회 기능이 실제로 준비되면 첫 화면은 아래처럼 바꾼다.

```text
기업과 인물을 빠르게 찾고 연결 관계를 확인하세요
```

Subcopy:

```text
금융위원회, DART, 주가 데이터를 기반으로 기업 개요·재무·공시·관계회사와 주요 인물의 역할·소속·관련 기업을 확인합니다.
```

검색창 placeholder:

```text
기업명, 인물명, 종목코드, 법인등록번호로 검색
```

## 4. 검색 IA

### 권장 검색 방식

인물 조회가 들어와도 첫 화면 검색창은 하나로 유지한다.

이유:

- 사용자는 먼저 이름을 입력하고 나중에 결과에서 구분하는 편이 자연스럽다.
- 영업/투자/리서치 업무에서는 기업명과 인물명을 번갈아 검색한다.
- 검색 진입점을 나누면 첫 화면이 복잡해진다.

### 검색 모드

결과 화면에서 type filter를 제공한다.

```text
전체 | 기업 | 인물
```

현재 단계:

- `전체 | 기업`만 제공하거나 type filter를 아예 숨긴다.
- 인물 기능이 없을 때 `인물` 탭을 disabled로 노출하지 않는다.

인물 기능 출시 단계:

- 검색 결과 상단에 segmented control을 추가한다.
- 기본값은 `전체`.
- 기업 결과와 인물 결과를 같은 리스트 안에 섞되, 타입 배지로 구분한다.

### 검색어 처리

기업 검색:

- 기업명
- 영문명
- 종목명
- 종목코드
- 법인등록번호
- 사업자등록번호

인물 검색:

- 이름
- 소속 기업
- 직책
- 과거/현재 임원명
- 주요주주명

주의:

- 사람 이름은 동명이인이 많다.
- 인물 검색 결과는 반드시 소속/직책/출처 기준을 함께 보여줘야 한다.

## 5. 첫 화면 디자인 계획

### 지금 적용할 방향

현재 첫 화면은 기업 조회에 집중한다.

권장:

- 메인 headline은 기업 중심으로 유지
- source rail에 `DART 공시/재무`, `금융위원회 기업기본정보`를 표시
- copy에는 아직 `인물`을 넣지 않는다.

### 미래 확장 여지를 남기는 방법

CSS/HTML naming:

- 새 공통 검색 영역 클래스는 `home-search`, `entity-search`처럼 generic 이름을 사용할 수 있다.
- 기존 `company-query` input id는 지금 유지해도 되지만, 다음 큰 리팩터링 때 `entity-query`로 바꾸는 계획을 둔다.

문구:

- 지금은 `기업명, 종목코드, 법인등록번호로 검색`
- 추후는 `기업명, 인물명, 종목코드, 법인등록번호로 검색`

CTA:

- 지금: `기업 검색하기`
- 추후: `통합 검색하기`

## 6. 검색 결과 리스트 디자인 계획

### 현재 기업 결과 row

기업 결과 row는 아래 정보를 가진다.

- 타입 배지: `기업`
- 회사명
- 영문명 또는 종목명
- 시장/상장 배지
- 업종
- 대표자
- 직원 수
- 데이터 상태
- CTA: `프로필 보기`, `비교 추가`

### 미래 인물 결과 row

인물 결과 row는 아래 정보를 가진다.

- 타입 배지: `인물`
- 이름
- 현재/최근 소속
- 직책
- 관련 기업 수
- 등장 출처: `DART 임원`, `주요주주`, `공시`, `보고서`
- 최근 기준일
- CTA: `인물 프로필 보기`, `관련 기업 보기`

예시:

```text
[인물] 홍길동
롯데이노베이트 · 대표이사
관련 기업 3개 · DART 2025 사업보고서 기준
```

### 기업/인물 혼합 결과 구분

검색 결과 row에는 타입 배지가 필요하다.

권장 배지:

- 기업: 파란색 계열
- 인물: 청록 또는 slate 계열

단, 배지 색만으로 구분하지 않는다.

필수 텍스트:

- `기업`
- `인물`

### 동명이인 UX

인물 검색 결과에는 아래 보조 정보가 필수다.

- 소속 기업명
- 직책
- 기준 보고서 또는 공시일
- 생년월 정보가 공개 데이터로 제공되는 경우에는 민감하게 노출하지 않고 `출처 보기` 안으로 넣는다.

기본 리스트에서 생년월을 노출하지 않는다.

이유:

- B2B 업무에 필요한 구분 정보는 소속/직책/관련 기업이다.
- 개인정보처럼 느껴지는 필드를 전면에 두면 신뢰감이 떨어진다.

## 7. 인물 상세 페이지 정보 구조

인물 상세는 기업 상세와 같은 시각 체계를 쓰되, 정보 순서는 다르게 둔다.

### 인물 상세 상단

1. 인물명
2. 현재/최근 소속
3. 직책
4. 데이터 기준
5. 출처 배지

예시:

```text
김경엽
롯데이노베이트 · 대표이사
DART 2025 사업보고서 기준 · 공개 공시 기반
```

### Executive Summary

인물 요약은 아래 항목으로 구성한다.

- 한줄 요약
- 현재 역할
- 관련 기업
- 최근 변동
- 확인할 점

AI 요약을 붙일 경우:

- 원천 데이터 기반
- 공개 공시 기반
- 추정 금지
- 투자 판단 아님

### 핵심 지표

기업의 key metrics와 다르게 인물은 관계 지표를 쓴다.

- 현재/최근 직책
- 관련 기업 수
- 등기임원 여부
- 주요주주 여부
- 최근 등장 공시일
- 기준 보고서

### 상세 섹션

권장 섹션:

1. 역할 및 소속
2. 경력/이력
3. 관련 기업
4. 지분/주요주주 정보
5. 공시 등장 이력
6. 데이터 출처

### 관계 그래프

인물 상세에서는 관계 그래프가 특히 유용하다.

노드:

- 인물
- 기업
- 직책
- 공시

엣지:

- `대표이사`
- `사내이사`
- `주요주주`
- `계열회사 임원`
- `공시 제출/등장`

초기 버전에서는 그래프를 만들지 않고, `관련 기업 리스트`와 `역할 태그`로 시작한다.

## 8. 기업 상세와 인물 상세의 공통 디자인 체계

### 공통 패턴

기업과 인물은 같은 페이지 골격을 공유한다.

```text
Hero
Executive Summary
Key Facts
Recent Signals
Detailed Sections
Trust Metadata
```

### 차이점

기업:

- 재무/공시/주가/관계회사 중심

인물:

- 역할/소속/관련 기업/등장 공시 중심

### 공통 컴포넌트 명명 원칙

앞으로 새로 만드는 컴포넌트는 가능하면 `entity` 접두어를 사용한다.

권장:

- `entity-hero`
- `entity-summary`
- `entity-key-metrics`
- `entity-result-row`
- `entity-type-badge`
- `entity-trust-meta`

현재 유지:

- 이미 존재하는 `company-*` 클래스는 당장 바꾸지 않는다.

이유:

- 현재 CSS/JS와 테스트가 안정적으로 동작하고 있다.
- 전체 rename은 회귀 위험이 크다.

추후 리팩터링:

- 인물 기능을 실제 구현할 때 `company-*`에서 공통화 가능한 부분만 `entity-*`로 분리한다.

## 9. 데이터 신뢰와 개인정보 UX

인물 조회는 기업 조회보다 신뢰/민감도 문제가 크다.

### 필수 표시

인물 상세에는 아래 문구 중 하나를 반드시 포함한다.

```text
공개 공시 및 보고서 기반 정보입니다.
```

또는:

```text
DART 정기보고서와 공개 공시에서 확인된 역할만 표시합니다.
```

### 표시하지 않을 것

초기 인물 조회에서는 아래 정보 노출을 피한다.

- 생년월 전면 노출
- 성별 전면 노출
- 주소
- 개인 연락처
- 원천 데이터에 없는 학력/경력 추정
- 언론/소문 기반 평판 요약

### 데이터 상태

인물 카드에는 상태를 명확히 둔다.

- `공시 기반`
- `보고서 기준`
- `최근 기준일`
- `부분 정보`
- `동명이인 주의`

## 10. 내비게이션 계획

### 현재

헤더:

- Profilage 정보
- API
- 문서
- OpenAPI

### 추천 현재 개선

기업 기능만 있을 때:

- `검색`
- `비교`
- `API`
- `데이터 문의`

### 인물 기능 출시 후

헤더:

- `검색`
- `기업`
- `인물`
- `비교`
- `API`
- `데이터 문의`

단, `기업`과 `인물`을 별도 페이지로 강하게 나누기보다 검색 결과 type filter로 연결한다.

### URL 구조

현재:

```text
/profile?crno=...
/compare?crno=...
```

미래:

```text
/company/profile?crno=...
/person/profile?person_id=...
/search?q=...&type=all
```

단기적으로 기존 URL은 유지한다.

마이그레이션 전략:

- `/profile?crno=...`는 계속 지원한다.
- 새 구조를 도입하면 `/company/profile?crno=...`로 canonical 링크를 제공한다.
- 기존 공유 링크가 깨지지 않게 redirect 또는 alias를 둔다.

## 11. 비교 기능과 인물

인물 조회가 들어오면 비교 기능은 세 가지로 나뉜다.

### 1단계: 기업 비교 유지

현재 비교는 기업끼리만 지원한다.

```text
기업 A vs 기업 B
```

### 2단계: 인물 비교는 보류

초기 인물 기능에서는 인물끼리 비교를 제공하지 않는다.

이유:

- 비교 기준이 기업보다 애매하다.
- 사용자 오해와 민감도 위험이 크다.
- 먼저 인물 상세 조회의 신뢰를 확보해야 한다.

### 3단계: 관계 비교 또는 네트워크 비교

나중에 제공 가능한 기능:

- 인물 A와 인물 B의 관련 기업 비교
- 한 기업과 연결된 주요 인물 목록
- 특정 인물과 특정 기업의 관계 경로

초기 CTA:

- `관련 기업 보기`
- `이 인물이 등장한 공시 보기`

제공하지 않을 CTA:

- `인물 비교에 추가`

### 통합 필터 원칙

- 데이터 출처
- 기준일

`기업`을 선택하면 기업 필터를 보여준다.

`인물`을 선택하면 인물 필터를 보여준다.

## 13. 지금 당장 디자인 계획에 반영할 것

현재 인물 기능을 만들지 않더라도, B2B UX 개선 계획에 아래 원칙을 추가한다.

### 카피

현재:

```text
기업 정보를 빠르게 찾고 비교하세요
```

유지한다.

미래 전환 후보:

```text
기업과 인물을 빠르게 찾고 연결 관계를 확인하세요
```

### 컴포넌트

새로 추가하는 공통 컴포넌트는 `entity` 이름을 우선 검토한다.

예:

- 검색 결과 row: `entity-result-row`
- 타입 배지: `entity-type-badge`
- trust meta: `entity-trust-meta`

단, 현재 계획의 PR 1~5에서는 기존 테스트 안정성을 위해 `company-*`와 `result-*`를 유지해도 된다.

### 결과 리스트

검색 결과 row 안에 타입 배지를 넣을 수 있는 자리만 확보한다.

현재:

```text
[기업] 롯데이노베이트
```

인물 출시 후:

```text
[인물] 김경엽
```

### 상세 페이지

상세 페이지의 상위 구조를 기업 전용이 아닌 공통 구조로 생각한다.

```text
Hero
Summary
Key Metrics
Recent Signals
Details
Trust
```

## 14. 미래 인물 조회 구현 PR 전략

인물 조회를 실제로 만들 때는 아래 순서가 좋다.

### PR 1: 인물 검색 데이터 모델과 API 초안

**Goal:** DART 임원/주요주주/직원 데이터에서 공개 인물 정보를 정규화한다.

예상 파일:

- Create: `app/services/person_profile.py`
- Modify: `app/api/company.py` 또는 Create: `app/api/person.py`
- Test: `tests/test_person_profile_api.py`

초기 API 후보:

```text
GET /api/person/search?name=...
GET /api/person/profile?person_id=...
```

### PR 2: 통합 검색 결과 타입 체계

**Goal:** 검색 결과가 기업과 인물 타입을 함께 표현할 수 있게 한다.

예상 파일:

- Modify: `app/static/app.js`
- Modify: `app/static/styles.css`
- Test: `tests/test_company_affiliate_api.py`

결과 item shape 후보:

```json
{
  "type": "person",
  "id": "person:dart:00126380:kim",
  "name": "김경엽",
  "subtitle": "롯데이노베이트 · 대표이사",
  "meta": ["DART 2025 사업보고서", "관련 기업 3개"],
  "href": "/person/profile?person_id=..."
}
```

### PR 3: 인물 상세 프로필

**Goal:** 사람 중심의 역할, 관련 기업, 공시 등장 이력을 보여준다.

예상 파일:

- Create: `app/static/person.html`
- Create: `app/static/person-page.js`
- Modify: `app/static/styles.css`
- Test: `tests/test_person_profile_api.py`

### PR 4: 기업 상세에서 관련 인물 섹션 추가

**Goal:** 기업 프로필에서 주요 인물 목록으로 확장한다.

예상 섹션:

- 대표자
- 등기임원
- 주요주주
- 최근 임원 관련 공시 등장 인물

### PR 5: 관계 탐색

**Goal:** 기업과 인물 간 연결 관계를 탐색한다.

초기 UI:

- 관련 기업 리스트
- 역할 태그
- 등장 공시 리스트

그래프 UI는 이 PR에서 만들지 않는다.

- [ ] 첫 화면 문구가 아직 제공하지 않는 인물 조회를 암시하지 않는다.

- [ ] trust metadata가 향후 인물 정보에도 그대로 적용 가능하다.

- [ ] 기업 프로필에서 관련 인물로 이동할 수 있다.
- [ ] 인물 프로필에서 관련 기업으로 이동할 수 있다.
- [ ] 인물 비교 CTA는 초기 버전에 없다.

## 17. 성공 기준

- 기업과 인물 사이를 자연스럽게 왕복 탐색할 수 있다.
